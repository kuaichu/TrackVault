# TrackVault 多平台架构改造方案 v1

## 1. 目标

TrackVault 后续如果要逐步覆盖网易云音乐、QQ 音乐 80% 以上的能力，继续沿用“每个平台单独堆一套页面逻辑”的方式，维护成本会迅速失控。

因此本项目后续采用：

- 统一领域模型
- 统一平台 provider 接口
- 平台差异收口到适配层

的演进方案。

这不是一次性推翻现有网易云代码，而是：

1. 先抽象统一层
2. 再把现有网易云能力逐步迁移进去
3. 最后接入 QQ provider

## 2. 为什么不继续纯 A 方案

如果继续纯“平台直连”模式，后面每增加一个平台，几乎都要重复写：

- 搜索
- 歌单
- 专辑
- 歌手
- 喜欢
- 歌词
- 播放地址
- 创建歌单 / 加歌

前端和后端都会越来越多：

```ts
if (platform === "netease") { ... }
if (platform === "qq") { ... }
```

这会导致：

- 代码重复
- 平台分叉严重
- 互转逻辑难写
- 新平台接入成本越来越高

## 3. 统一层设计

### 3.1 统一领域模型

先抽象出跨平台通用对象：

- `UnifiedTrack`
- `UnifiedArtist`
- `UnifiedAlbum`
- `UnifiedPlaylist`
- `UnifiedSearchResult`

这些对象不关心底层是网易云还是 QQ，只表达业务语义。

### 3.2 统一 provider 能力

后续平台能力以 provider 接口暴露，例如：

- `searchTracks`
- `getPlaylistList`
- `getPlaylistDetail`
- `getArtistProfile`
- `getAlbumProfile`
- `getLyrics`
- `getPlayableResource`
- `createPlaylist`
- `addTracksToPlaylist`

### 3.3 平台适配器

每个平台单独维护一个适配器：

- `netease-adapter`
- `qq-adapter`

负责：

- 把平台原始返回值转换成统一领域模型
- 处理平台特有字段与音质信息
- 维护平台内歌曲 ID、歌手 ID、专辑 ID 的标准化

## 4. 当前分阶段施工顺序

### 第一阶段：打统一骨架

目标：

- 新增统一领域模型文件
- 新增网易云适配器
- 先把“搜索”迁移到统一层

特点：

- 不改现有页面行为
- 不动 UI 布局
- 只做后端内部收口

### 第二阶段：迁移网易云内容型接口

目标：

- 发现音乐
- 每日推荐
- 歌单详情
- 云盘歌曲
- 歌手页热门歌曲
- 专辑页歌曲

都逐步改成先输出 `UnifiedTrack`，再转换成当前前端 `Song`

### 第三阶段：定义 QQ provider

目标：

- 先接 QQ 搜索
- 再接 QQ 歌单列表与歌单详情
- 再接 QQ 创建歌单 / 加歌

### 第四阶段：实现歌单互转

目标：

- 网易云 -> QQ
- QQ -> 网易云

核心链路：

1. 读取源歌单
2. 转成统一曲目结构
3. 到目标平台搜索匹配
4. 新建目标歌单
5. 批量加歌
6. 输出迁移报告

## 5. 匹配策略建议

歌单互转不可能只靠平台 songId，需要做“歌曲相似匹配”：

- 歌名
- 主歌手
- 专辑名
- 时长

建议后续采用评分制：

- 歌名完全一致：高权重
- 主歌手命中：高权重
- 时长接近：中权重
- 专辑命中：中权重

最后选得分最高的一首作为目标平台候选。

## 6. 当前这一步会做什么

本轮开始施工时，只做最简单也最稳的一步：

- 落地统一平台类型
- 落地 `netease-adapter`
- 把现有网易云搜索先接入统一层

这样下一步接 QQ 搜索时，就不需要直接碰现有页面，只要做一个 `qq-adapter + qq-provider` 即可。

## 7. 当前已落地进度

截至当前版本，统一层已经覆盖：

- 网易云搜索
- 发现音乐
- 每日推荐
- 歌单歌曲
- 云盘歌曲
- 歌手页热门歌曲
- 专辑页歌曲

并且已经新增：

- `qq-search-provider`

当前 QQ 能力先落在后端搜索层，可通过：

- `GET /api/search?q=关键词&platform=qq`

拿到统一后的歌曲结构。

这一步暂时还没有把 QQ 搜索直接暴露到当前前端页面，因为播放器、下载、歌词、歌手页、专辑页等链路仍然是网易云主导；等 QQ 的媒体与详情能力继续补齐后，再把平台切换正式接到页面上。
